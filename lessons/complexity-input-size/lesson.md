# 복잡도와 입력 크기 감각

알고리즘 문제를 풀 때 가장 먼저 해야 하는 일은 "어떤 풀이가 시간 안에 가능한가"를 가늠하는 것입니다. 아이디어가 맞아도 입력 크기에 비해 복잡도가 크면 통과할 수 없습니다. 반대로 입력이 작다면 복잡한 자료구조보다 단순한 완전탐색이 더 안전할 수 있습니다.

이 레슨은 시간 복잡도 표기법을 외우는 데서 끝나지 않고, `n <= 2,000`, `n <= 100,000`, `n <= 10^6` 같은 조건을 보고 가능한 풀이 후보를 좁히는 감각을 다룹니다.

## 0. 선수 지식과 이어지는 레슨

- 선수 지식: 기본 사칙연산과 반복문
- 함께 보면 좋은 레슨: 정렬 알고리즘, 이분 탐색, 동적 계획법
- 다음에 볼 레슨: 대회용 C++ 기본기, 정렬 알고리즘, 누적합과 차분 배열

## 1. 시간 제한을 연산 횟수로 바꾸기

보통 1초에 몇 번의 연산이 가능한지는 언어, 서버, 연산 종류에 따라 달라집니다. 그래도 처음 풀이를 고를 때는 아래처럼 거칠게 생각하면 충분합니다.

| 감각 | 의미 |
| --- | --- |
| `10^6` 정도 | 대부분 여유 있음 |
| `10^7` 정도 | 보통 안전한 편 |
| `10^8` 정도 | C++에서도 구현과 상수에 따라 위험 |
| `10^9` 이상 | 거의 항상 다른 풀이가 필요 |

이 숫자는 정답이 아니라 필터입니다. `10^8`번의 단순 덧셈과 `10^8`번의 map 탐색, 문자열 비교, heap 연산은 체감 시간이 다릅니다. 복잡도 판단은 "가능성이 있는 후보를 남기고, 명백히 느린 후보를 버리는" 용도로 씁니다.

## 2. 입력 크기에서 가능한 복잡도 찾기

입력 크기를 보면 먼저 아래 표처럼 후보를 좁힙니다.

| 입력 크기 | 자주 가능한 복잡도 | 대표 풀이 감각 |
| ---: | --- | --- |
| `n <= 20` | `O(2^n * n)`, `O(n!)` 일부 | 비트마스크, 백트래킹, 작은 TSP |
| `n <= 100` | `O(n^3)` | Floyd-Warshall, 구간 DP |
| `n <= 2,000` | `O(n^2)` | 모든 쌍, 기본 DP |
| `n <= 100,000` | `O(n log n)`, `O(n sqrt n)` 일부 | 정렬, 이분 탐색, heap, Fenwick Tree |
| `n <= 1,000,000` | `O(n)`, `O(n log n)` 일부 | 한 번 훑기, 누적합, counting |
| `n <= 10^9` 이상 | `O(log n)`, `O(sqrt n)`, 수식 | 이분 탐색, 빠른 거듭제곱, 수학 |

예를 들어 `n <= 100,000`인데 모든 쌍을 보는 `O(n^2)` 풀이를 떠올렸다면, 대략 `10^10`번의 비교가 필요합니다. 이 경우 정렬 후 한 번 훑기, 이분 탐색, 자료구조, 누적합처럼 반복문 하나를 줄이는 구조를 찾아야 합니다.

## 3. 반복문을 보고 직접 세기

Big-O 표기만 보지 말고 코드에서 실제 반복 횟수를 세는 습관이 중요합니다.

```cpp
long long countPairs(int n) {
    long long count = 0;
    for (int i = 0; i < n; ++i) {
        for (int j = i + 1; j < n; ++j) {
            count++;
        }
    }
    return count;
}
```

이 코드는 정확히 `n * (n - 1) / 2`번 반복합니다. Big-O로는 `O(n^2)`입니다. `n = 2,000`이면 약 200만 번이라 괜찮지만, `n = 100,000`이면 약 50억 번이라 어렵습니다.

중첩 반복문이 항상 `O(n^2)`인 것은 아닙니다.

```cpp
int moves = 0;
for (int left = 0, right = 0; left < n; ++left) {
    while (right < n && canExtend(left, right)) {
        right++;
        moves++;
    }
}
```

겉으로는 `for`와 `while`이 중첩되어 있지만, `right`가 전체 실행 동안 `0`에서 `n`까지 한 번만 증가한다면 전체는 `O(n)`입니다. 투 포인터와 슬라이딩 윈도우에서 자주 쓰는 감각입니다.

## 4. 로그와 제곱근 감각

`log n`은 아주 천천히 증가합니다. `n`이 10억이어도 `log2(n)`은 약 30입니다. 그래서 정렬의 `O(n log n)`, 이분 탐색의 `O(log n)`, heap 연산의 `O(log n)`은 큰 입력에서도 자주 버팁니다.

반대로 `sqrt n`은 `n`보다는 작지만 생각보다 큽니다.

| `n` | `sqrt(n)` |
| ---: | ---: |
| `10^4` | `100` |
| `10^6` | `1,000` |
| `10^9` | 약 `31,623` |

약수 열거, sqrt decomposition, Mo's Algorithm 같은 풀이에서는 `sqrt n`이 등장합니다. `n sqrt n`이 가능한지 판단할 때는 실제 곱을 계산해 봐야 합니다.

## 5. 메모리도 같이 본다

시간만 맞아도 메모리가 넘치면 실패합니다. C++에서 자주 쓰는 타입 크기는 아래처럼 잡으면 됩니다.

| 타입 | 대략적인 크기 |
| --- | ---: |
| `int` | 4 bytes |
| `long long` | 8 bytes |
| `double` | 8 bytes |
| `pair<int, int>` | 8 bytes |

예를 들어 `int dp[5000][5000]`은 원소가 2,500만 개입니다. `int`는 4 bytes이므로 약 100MB가 필요합니다. 여기에 다른 배열과 실행 환경 오버헤드까지 더해지면 제한에 걸릴 수 있습니다.

```cpp
long long bytesForIntTable(long long rows, long long cols) {
    return rows * cols * 4LL;
}
```

배열 크기를 볼 때도 overflow를 피하려고 `long long`으로 곱하는 습관이 좋습니다.

## 6. 복잡도 줄이는 대표 방향

느린 풀이를 빠르게 바꾸는 방향은 대체로 반복문 하나를 구조로 대체하는 것입니다.

| 느린 형태 | 줄이는 방향 | 이어지는 레슨 |
| --- | --- | --- |
| 모든 쌍을 직접 비교 | 정렬 후 인접한 것만 보기 | 정렬 |
| 매번 구간 합을 다시 계산 | 누적합으로 구간 합 `O(1)` | 누적합 |
| 답을 하나씩 시도 | 단조성을 찾아 이분 탐색 | 이분 탐색 |
| 매번 최솟값을 선형 탐색 | heap 또는 segment tree | 우선순위 큐, Segment Tree |
| 같은 부분문제 반복 | DP 테이블에 저장 | 동적 계획법 |
| 큰 좌표 범위 직접 배열 | 실제 등장 값만 압축 | 좌표 압축 |

문제 조건을 읽을 때는 "내가 지금 반복하는 대상이 꼭 전부 필요한가"를 계속 물어봐야 합니다.

## 7. 자주 하는 실수

- 시간 제한만 보고 입력 크기를 놓친다.
- `O(n^2)`에서 `n = 100,000`을 시도한다.
- 중첩 반복문을 무조건 `O(n^2)`로 보거나, 반대로 모든 중첩 반복문을 과소평가한다.
- `map`, `set`, `priority_queue`의 `log n` 비용을 상수처럼 생각한다.
- 메모리 제한을 보지 않고 큰 2차원 배열을 만든다.
- `int` 곱셈으로 배열 크기나 경우의 수를 계산해 overflow를 만든다.
- 문제의 테스트 케이스 개수 `T`를 복잡도에 곱하지 않는다.

## 8. 문제를 볼 때 체크할 조건

1. 입력 크기 `n`, `m`, `q`, `T`가 각각 얼마인가?
2. 내가 생각한 풀이의 반복 횟수를 대략 곱하면 얼마인가?
3. `O(n^2)`, `O(n log n)`, `O(n)` 중 어디까지 가능한가?
4. 배열과 DP 테이블의 메모리 사용량은 몇 MB인가?
5. 값의 범위가 큰지, 실제 등장하는 값의 개수가 작은지 구분했는가?
6. 정렬, 누적합, 이분 탐색, 자료구조로 반복문 하나를 줄일 수 있는가?

## 대표 문제로 연결하기

### 문제에서 보이는 신호

| 신호 | 판단 |
| --- | --- |
| 입력 크기 | `n <= 2,000`이면 모든 쌍, `n <= 100,000`이면 `O(n log n)` 이하를 먼저 의심 |
| 키워드 | "모든 쌍", "구간 합", "최대/최소", "질의가 많다", "값 범위가 크다" |
| 자주 나오는 변형 | 테스트 케이스가 여러 개라 전체 입력 합을 봐야 하는 문제, `n`과 `m`을 모두 곱해야 하는 문제 |

### 풀이 흐름

1. 입력 제한을 표에 대입해 가능한 복잡도 범위를 잡는다.
2. 가장 단순한 풀이의 반복 횟수를 직접 센다.
3. 불가능하면 정렬, 누적합, 이분 탐색, DP, 자료구조 중 어떤 구조가 반복을 줄이는지 찾는다.
4. 메모리 제한에 맞게 배열 크기와 타입을 계산한다.
5. 테스트 케이스 수까지 곱해 최종 비용을 다시 확인한다.

### 자주 틀리는 지점

- `n`만 보고 `q`개의 질의를 빼먹는다.
- 각 테스트 케이스의 `n`이 작아 보여도 `T * n^2`이 커지는 경우를 놓친다.
- `long long`이 필요한 누적합을 `int`로 계산한다.
- `O(n log n)`이라고 적었지만 내부에서 문자열 비교나 map 탐색이 추가로 들어간다.
