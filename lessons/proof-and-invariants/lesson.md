# Proof와 Invariant

알고리즘 풀이에서 증명은 별도의 이론 과제가 아니라, "왜 이 코드가 모든 입력에서 맞는가"를 확인하는 과정입니다. 특히 그리디, 이분 탐색, DP, 자료구조 문제는 구현보다 전제와 불변식이 더 중요할 때가 많습니다.

이 레슨은 풀이를 설명하고 검증할 때 자주 쓰는 네 가지 도구를 다룹니다.

1. Invariant: 반복 중 항상 유지되는 조건
2. Exchange argument: 더 나은 선택으로 바꿔도 손해가 없다는 논증
3. Monotonicity: 이분 탐색이 가능한 참/거짓 경계
4. Counterexample search: 틀린 직감을 깨는 작은 입력 찾기

## 0. 선수 지식과 이어지는 레슨

- 선수 지식: 복잡도 감각, 그리디, 이분 탐색, 동적 계획법
- 함께 보면 좋은 레슨: Testing과 Stress Test, 그리디 알고리즘
- 다음에 볼 레슨: 고급 최적화 DP, matroid/flow duality 같은 증명형 주제

## 1. Invariant

Invariant는 알고리즘이 진행되는 동안 항상 참이어야 하는 조건입니다. 반복문이나 자료구조를 이해할 때 특히 중요합니다.

예를 들어 이분 탐색에서는 아래 invariant를 둘 수 있습니다.

```text
lo는 아직 불가능한 값
hi는 가능한 값
정답은 항상 (lo, hi] 안에 있다
```

반복할 때마다 이 조건이 유지되면, 마지막에 `hi`가 최소 가능한 값이라는 결론을 낼 수 있습니다.

## 2. 이분 탐색의 증명 틀

이분 탐색 문제를 설명할 때는 코드보다 먼저 단조성을 말해야 합니다.

1. 어떤 값 `x`에 대해 `check(x)`를 정의한다.
2. `check(x)`가 참이면 더 큰 값도 참인지, 또는 더 작은 값도 참인지 확인한다.
3. 경계의 어느 쪽을 답으로 출력할지 정한다.
4. 반복문의 invariant를 유지한다.

```cpp compile-check
#include <vector>
using namespace std;

bool canMakeAtLeast(const vector<int>& a, long long target, int need) {
    int count = 0;
    for (int value : a) {
        if (value >= target) {
            ++count;
        }
    }
    return count >= need;
}

long long maxFeasibleValue(const vector<int>& a, int need) {
    long long lo = 0;
    long long hi = 1000000001LL;
    while (lo + 1 < hi) {
        long long mid = (lo + hi) / 2;
        if (canMakeAtLeast(a, mid, need)) {
            lo = mid;
        } else {
            hi = mid;
        }
    }
    return lo;
}
```

위 코드는 `lo`가 가능한 값, `hi`가 불가능한 값이라는 invariant를 둡니다. 그래서 마지막에 가능한 최댓값인 `lo`를 반환합니다.

## 3. Exchange Argument

Exchange argument는 그리디 증명에서 자주 씁니다.

전형적인 형태는 아래와 같습니다.

```text
어떤 최적해 O가 있다고 하자.
내 그리디가 고른 선택 g가 O에 없다면,
O의 어떤 선택 x를 g로 바꿔도 해는 여전히 가능하고 값은 나빠지지 않는다.
그러면 g를 포함하는 최적해가 존재한다.
따라서 그리디의 첫 선택은 안전하다.
```

회의실 배정에서 "가장 빨리 끝나는 회의를 먼저 고른다"는 선택이 대표적입니다. 어떤 최적해가 첫 회의로 더 늦게 끝나는 회의를 골랐다면, 그것을 더 빨리 끝나는 회의로 바꿔도 이후 선택 가능성은 줄어들지 않습니다.

## 4. DP 증명 틀

DP는 상태 정의와 전이의 완전성이 핵심입니다.

1. `dp[state]`가 무엇을 의미하는지 한 문장으로 정의한다.
2. 초기값이 정의와 맞는지 확인한다.
3. 모든 가능한 마지막 선택이 전이에 포함되는지 확인한다.
4. 같은 해를 여러 번 세는지, 빠뜨리는지 확인한다.
5. 계산 순서가 의존성을 만족하는지 확인한다.

DP에서 가장 흔한 오류는 "전이가 그럴듯하지만 상태 정의와 맞지 않는" 경우입니다. 상태 정의를 먼저 고정하지 않으면 증명도 구현도 흔들립니다.

## 5. Counterexample 찾기

증명을 못 하겠다면 작은 반례를 찾는 것이 좋습니다. 반례는 보통 아래에서 나옵니다.

| 직감 | 반례 후보 |
| --- | --- |
| 가장 큰 것을 먼저 고른다 | 큰 것 하나보다 작은 것 여러 개가 좋은 경우 |
| 가까운 것을 먼저 연결한다 | 나중에 더 큰 비용을 만드는 경우 |
| 정렬 후 인접한 것만 보면 된다 | 전역 조건이 숨어 있는 경우 |
| 각 단계의 최선이 전체 최선이다 | 선택이 미래 가능성을 줄이는 경우 |

작은 입력에서 가능한 모든 해를 나열해 보면 직감이 맞는지 빨리 확인할 수 있습니다. 이 과정은 stress test와도 연결됩니다.

## 6. 자료구조 Invariant

자료구조 구현에서도 invariant가 필요합니다.

| 자료구조 | 대표 invariant |
| --- | --- |
| Heap | 부모가 자식보다 우선순위가 높다 |
| Union-Find | parent를 따라가면 대표 원소에 도달한다 |
| Segment Tree | 노드는 담당 구간의 merge 결과를 저장한다 |
| Monotonic Queue | 큐 안의 후보가 단조 순서를 유지한다 |
| Two Pointers | 포인터가 되돌아가지 않는다 |

버그가 났을 때는 출력값만 보지 말고 invariant가 언제 깨지는지 확인해야 합니다.

## 7. 증명 스케치 템플릿

풀이 설명을 쓸 때는 아래 템플릿이 유용합니다.

```text
정의:
  이 변수/상태/자료구조가 의미하는 것

Invariant:
  반복문 시작 또는 종료 시 항상 참인 조건

유지:
  한 단계 실행 후에도 invariant가 유지되는 이유

종료:
  반복이 끝났을 때 invariant가 답을 보장하는 이유

복잡도:
  각 원소/간선/상태가 몇 번 처리되는지
```

이 템플릿은 완전한 수학 증명이 아니어도, 풀이의 빈틈을 찾는 데 충분히 도움이 됩니다.

## 8. 자주 하는 실수

| 실수 | 결과 | 확인 방법 |
| --- | --- | --- |
| 단조성을 확인하지 않고 이분 탐색 | 경계가 의미 없음 | `false...true` 또는 `true...false` 구조 확인 |
| 그리디 선택을 예시로만 설득 | 숨은 반례 | exchange argument 시도 |
| DP 상태 정의가 모호함 | 전이 누락/중복 | `dp[state]` 한 문장 정의 |
| invariant가 시작 시점에만 참 | 반복 후 깨짐 | 유지 단계 확인 |
| 종료 조건과 답 위치 불일치 | off-by-one | 마지막 invariant로 반환값 검증 |
| 반례 탐색을 하지 않음 | 직감 오답 | 작은 입력 완전탐색 |

## 9. 문제를 볼 때 체크할 조건

1. 내 풀이의 핵심 선택이 항상 안전한가?
2. 반복 중 유지해야 하는 invariant가 무엇인가?
3. 이분 탐색을 쓴다면 단조성이 정확히 무엇인가?
4. DP 상태가 한 문장으로 정의되는가?
5. 작은 반례를 손으로 만들 수 있는가?
6. 종료 시점에 반환하는 값이 invariant와 맞는가?

정리하면, proof는 풀이를 꾸미는 문장이 아니라 구현 전에 버그를 줄이는 도구입니다. 안전한 선택, 유지되는 조건, 단조 경계, 반례 후보를 명시하면 코드도 훨씬 안정됩니다.
